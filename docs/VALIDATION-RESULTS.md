## Live integration proof — 2026-06-28

The six tools were exercised against a **real** Oracle, not a fake cursor.

- **Substrate:** `gvenzl/oracle-xe:21-slim` (`XEPDB1`), brought up via
  `docker compose -f docker-compose.oracle.yml up -d --wait`, seeded at first
  boot from `tests/fixtures/seed_schema.sql`.
- **Command:** env-gated
  `COGNIC_RUN_ORACLE_INTEGRATION=1 COGNIC_ORACLE_DSN=localhost:1521/XEPDB1
  COGNIC_ORACLE_USER=cognic COGNIC_ORACLE_PASSWORD=cognic_dev_only
  pytest tests/integration -m oracle -q`.
- **Result:** **6 / 6 passed (~7.8 s)** against the live database — one test per
  tool (`list_schemas`, `list_tables`, `describe_table`, `find_columns`,
  `list_relationships`, `get_constraints`).
- **Seed pattern validated:** the seed runs admin-side
  (`ALTER SESSION SET CONTAINER` + fully-qualified `cognic.*` object DDL); the
  tools then connect **as** the seed owner so the `ALL_*` data-dictionary views
  surface the account's own objects (self-owned visibility).
- **Teardown confirmed:** `docker compose -f docker-compose.oracle.yml down -v`
  removed the container and its volume cleanly.

This is the local live-integration bar. The deployed acceptance bar (Proof-2 /
M3-E2c — install the released signed pack into a deployed AgentOS, reach
`discovery_status = auth_ready`, and `call_tool(describe_table)` over the
override-pinned ClusterIP) is a separate follow-on; the M3 milestone checkbox
flips only when that runs green.

## `sign` / `verify` proof — 2026-06-28

Built, signed (real cosign — no `--dev-mode-skip-cosign`), and verified GREEN
against a dedicated keypair, on this host.

**Toolchain:** cosign `v3.0.6` · syft `1.44.0` · grype `0.112.0` · pip-licenses
`5.5.5` (`uv tool`); the `agentos` CLI from the `@v0.0.2` dev extra.

**Signing identity — dev-grade release-proof key, NOT a production trust root:**

- Private key: `~/.cognic/signing/cognic-tool-oracle-schema/v0.1.0/cosign.key`
  (empty passphrase, local custody, **never committed**).
- Public trust root: committed at `cosign.pub` (repo root), sha256
  `43c33fbe7f4b16683d47886b81cb1b9684495cbb9a92989b10f5b8cd72ba2e78`.
- Production deployments use keyless-OIDC (Fulcio) or an HSM/KMS-custodied key,
  not a local empty-passphrase key. This key exists only to prove the
  sign → verify path for the M3-E2b milestone. Establishing a pack's first trust
  root is a human-only decision, authorized by the maintainer for this proof.

**Build:** `uv build --wheel` →
`dist/cognic_tool_oracle_schema-0.1.0-py3-none-any.whl`, sha256
`4ed1a44773696429acf6bd5e88d91fa966ab9c4a0a3dc80925bac179883b1beb`.

**Sign:** `COGNIC_SIGNING_KEY_PATH=<key> COSIGN_PASSWORD="" agentos sign --bundle .`
→ `sign-bundle: PASS`. The 7-attestation bundle (build output under
`attestations/`, **not** source-controlled — Model Y): `cosign.sig`,
`bundle.sigstore`, `sbom.cdx.json` (CycloneDX/syft, ~740 KB),
`slsa-provenance.intoto.json`, `intoto-layout.json`, `vuln-scan.json` (grype),
`license-audit.json` (pip-licenses).

**Verify:** `agentos verify --trust-root cosign.pub .` → `verify: PASS` — all 7
attestations verified (offline trust gate, 11 steps incl. the isolated
load-probe), anchored to the **explicit committed `cosign.pub`**, never the
implicit `Settings.signing_trust_root_path` default.

**Source-manifest integrity — the `blob_path` nuance:** `agentos sign --bundle`
rewrites `cognic-pack-manifest.toml` in place (strips every comment, reformats
arrays, and adds `[supply_chain].blob_path =
"cognic_tool_oracle_schema-0.1.0-py3-none-any.whl"`). That `blob_path` is local
bundle bookkeeping, **not** a required verifier field — `verify --trust-root
cosign.pub .` passes GREEN against the **restored clean source manifest**
(comments intact, no `blob_path`). The clean manifest is therefore kept as
source and the generated `blob_path` is recorded here as evidence. Kernel
follow-up candidate: `agentos sign` should not destructively rewrite the source
manifest.

**Release-artifact model (Model Y):** the repo source-controls the public trust
root (`cosign.pub`) + this proof record; the wheel and the `attestations/`
bundle are build outputs regenerated by `agentos sign` on demand (wired in
`.github/workflows/sign-and-publish.yml`), not committed.
